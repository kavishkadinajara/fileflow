3. BLOCKERS / OPEN QUESTIONS

Q: Should the benchmark runner be gated behind an env flag so it doesn't run in CI?
A: Not in CI at all for now — it's a research tool, not a regression test. Add a note
   in README that it requires both servers running.

Q: File size for fixture documents — should we aim for realistic sizes?
A: Yes. Mix of short (1–2 pages), medium (5–10 pages), and long (15+ pages) documents
   gives better signal. Short documents are edge cases for the chunking logic; long
   documents stress-test latency.

Q: What about non-English documents?
A: Out of scope for v1 of ConvertBench. Note it in the dataset README as a known gap.

Q: Do we commit the generated .docx and .pdf fixtures to the repo?
A: No — binary files bloat git history badly. generate_fixtures.py + a README explaining
   how to regenerate them is the right approach. The .md and .html sources are the ground
   truth; everything else is derived.

4. INFRASTRUCTURE

Python backend deployment options discussed:
- Current: manual uvicorn on dev machine. Not viable for production.
- Option A: Docker container on same VPS as Next.js. Simple, cheap.
- Option B: Render.com free tier. Zero-config, but cold starts on free tier could
  reintroduce the model warm-up problem.
- Option C: Modal.com — serverless Python with GPU support for the sentence-transformer
  inference. Would speed up scoring significantly. $0.04/hr on-demand.

Decision: Implement Option A (Docker) for self-hosted tier. Evaluate Modal for the
public cloud version. Add to Q2 roadmap.
